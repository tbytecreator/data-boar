# Source note — supply chain trust and “governance vs reality”

**Português (Brasil):** [SUPPLY_CHAIN_AND_TRUST_SIGNALS.pt_BR.md](SUPPLY_CHAIN_AND_TRUST_SIGNALS.pt_BR.md) — short **deferred follow-ups** mirror; full note remains **English** below.

Primary links (examples of the pattern, not endorsements of every claim):

- [Shadow AI: governance reports vs operational reality](https://glazar.com.br/shadow-ai-o-relatorio-de-governanca-diz-que-esta-tudo-certo-e-acredite-nao-esta/) (Glazar, 2026) — gap between **reported** visibility and **unauthorized** AI use; non-human identities, OAuth sprawl, agents, MCP-style exposure.
- **Open VSX / marketplace class of failure (2026):** public reporting (e.g. security community / LinkedIn summaries) on **fail-open** behaviour where **pre-publish scanning** for extensions could be **bypassed**, allowing **malicious VS Code–style extensions** into a registry. Treat as a reminder that **trust boundaries** for **IDE extensions** and **package registries** are not guaranteed by a green dashboard alone.

Why this matters for Data Boar:

- **Buyers** care whether we could be an **unintentional lateral-movement** or **supply-chain** hop. Honest scope in [SECURITY.md](../../SECURITY.md), [COMPLIANCE_AND_LEGAL.md](../../COMPLIANCE_AND_LEGAL.md), and tests beats over-claiming “total governance.”
- **Shadow AI** is mostly an **organizational** control problem; our product **supports evidence** for **data** discovery, not inventory of every **AI agent** or **extension** in the customer estate.

How we consume it (repository and pipeline):

- **Python supply chain:** committed **`uv.lock`**, **`pip-audit`** in CI, **Dependabot** for pip + Actions — see root **SECURITY.md** and [ADR 0003](../../adr/0003-sbom-roadmap-cyclonedx-then-syft.md) (SBOM roadmap for **inventory / IR**).
- **GitHub Actions:** minimal **`permissions`** where configured; **CI / CodeQL / Semgrep / slack-ops-digest** pin third-party Actions to **full commit SHAs** (tag in YAML comment); **`astral-sh/setup-uv`** pins the **uv** CLI to a **semver** in **`ci.yml`** (not `latest`). **Dependabot** (github-actions) proposes SHA bumps—review release notes before merge. **Why and limits:** [ADR 0005](../../adr/0005-ci-github-actions-supply-chain-pins.md) (not a zero-day or laptop-tooling guarantee).
- **March 2026 — Trivy ecosystem compromise:** Public advisories describe **malicious updates** to **Trivy** binaries/images and to **`trivy-action` / `setup-trivy`** (including **tag** manipulation), stealing **CI secrets** while scans looked legitimate. **This repo does not run Trivy in workflows** (no `trivy` Action in `.github/workflows/`). **Mitigations that apply here:** SHA-pinned **`uses:`** for the Actions we **do** use reduces **moving-tag** risk; **does not** remove risk if a **pinned** commit is later found malicious — follow **vendor advisories** and **rotate credentials** if a tool you use is implicated. References: [GHSA-69fq-xp46-6x23](https://github.com/advisories/GHSA-69fq-xp46-6x23), [Microsoft guidance](https://www.microsoft.com/en-us/security/blog/2026/03/24/detecting-investigating-defending-against-the-trivy-supply-chain-compromise/).
- **Operator / maintainer workstation:** extension marketplaces, MCP servers, and “helpful” automation live **outside** this repo’s CI — document habits under **gitignored** `docs/private/` if needed; do not put **live** tool inventories in **tracked** Markdown.
- **Lab-op / homelab monitoring (learning):** Optional SIEM-style stack study (e.g. **Wazuh** official docs) and **NIST CSF / CIS** vocabulary for buyer conversations — [WAZUH_NIST_CIS_LABOP_ALIGNMENT.md](WAZUH_NIST_CIS_LABOP_ALIGNMENT.md) · [pt-BR](WAZUH_NIST_CIS_LABOP_ALIGNMENT.pt_BR.md). Data Boar **does not** ship Wazuh; lab monitoring is **operator** scope.
- **Enterprise DB operations vs GRC workbooks:** Map **backup/restore discipline**, patching culture, and generic **cyber/GRC template** rows to **honest** Data Boar **export** roles — [ENTERPRISE_DB_OPS_AND_GRC_EVIDENCE.md](ENTERPRISE_DB_OPS_AND_GRC_EVIDENCE.md) · [pt-BR](ENTERPRISE_DB_OPS_AND_GRC_EVIDENCE.pt_BR.md).

Watch-outs:

- **CI green** and **compliance prose** do not prove **zero** shadow tooling on developer laptops.
- **pip-audit** catches **known** vulnerable packages, not every **malicious** or **typosquat** scenario.
- **“Trusted” scanners in CI** can still become a **delivery channel** if their **Action tags**, **binaries**, or **containers** are compromised — treat like any other third-party dependency.
- Keep **product scope** clear: we are **not** an enterprise “AI governance platform” unless we build and document that explicitly.

Deferred follow-ups (process and posture — **no** new mandatory operator commands yet):

- **GitHub settings:** Revisit **fork PR** workflow **approvals**, **secret** exposure rules, and **review** discipline for `.github/workflows` changes on a calendar basis — see GitHub docs [Approving workflow runs from public forks](https://docs.github.com/en/actions/managing-workflow-runs/approving-workflow-runs-from-public-forks), Security Lab [Preventing pwn requests](https://securitylab.github.com/research/github-actions-preventing-pwn-requests/), and [CloudQuery: PRs merged without review](https://www.cloudquery.io/blog/how-to-use-the-github-plugin-to-find-pull-requests-that-were-merged-without-review) as a **governance hygiene** reminder.
- **`workflow_run`:** This repository uses **`workflow_run`** only for [Slack CI failure notify](../../.github/workflows/slack-ci-failure-notify.yml) (narrow **`permissions`**, no checkout). Any **additional** `workflow_run` jobs should stay minimal and be reviewed when ecosystem incidents highlight “downstream” workflow risk.
- **Ecosystem incident mode:** When a **widely used** scanner, Action, or registry is implicated, treat it as a **structured check**: confirm **usage**, read **GHSA/vendor** guidance, align with [ADR 0005](../../adr/0005-ci-github-actions-supply-chain-pins.md) **SHA pin** practice, and **rotate** CI-exposed secrets if warranted. Example readouts: [GHSA-69fq-xp46-6x23](https://github.com/advisories/GHSA-69fq-xp46-6x23), [Microsoft (Trivy)](https://www.microsoft.com/en-us/security/blog/2026/03/24/detecting-investigating-defending-against-the-trivy-supply-chain-compromise/), [CrowdStrike (trivy-action)](https://www.crowdstrike.com/en-us/blog/from-scanner-to-stealer-inside-the-trivy-action-supply-chain-compromise/), [Aikido (Shai-Hulud)](https://www.aikido.dev/blog/github-actions-incident-shai-hulud-supply-chain-attack), [GitGuardian (Shai-Hulud 2.0)](https://blog.gitguardian.com/shai-hulud-2/); maintain **KEV/advisory** habit per [CISA KEV note](CISA_KEV_AND_ADVISORIES.md). **Tracked backlog row:** [WORKFLOW_DEFERRED_FOLLOWUPS.md](../WORKFLOW_DEFERRED_FOLLOWUPS.md) (same items in the “Still to deepen” table).
- **Recovery discipline (lab-op / homelab):** On a **calendar** basis, validate that **backup and restore** actually work for hosts that matter for central monitoring (e.g. goals around **Wazuh** in lab-op). This is **operator process**, not a new mandatory repo command; it maps to **Recover** in [NIST CSF](https://www.nist.gov/cyberframework) vocabulary. Learning anchor (official Wazuh doc index + NIST/CIS alignment): [WAZUH_NIST_CIS_LABOP_ALIGNMENT.md](WAZUH_NIST_CIS_LABOP_ALIGNMENT.md) · [pt-BR](WAZUH_NIST_CIS_LABOP_ALIGNMENT.pt_BR.md).
- **Framework vocabulary for buyers:** Use **NIST CSF** and **CIS Controls** as **positioning** language for controls the **repository and product** already implement; do **not** imply certification or full-framework coverage without a formal program. Same learning note as above.
