# Release 1.6.0

**Release date:** 2026-03

Feature release: compressed-files scanning is wired end-to-end (filesystem + shares + dashboard/API), CNPJ alphanumeric detection is supported via config flag and overrides, and the dashboard scan flow is more reliable and observable. Version, docs, and tests are aligned with these capabilities.

## Highlights

- **Compressed files plan completed (Tier 1 + Tier 2 archives):**
- Filesystem connector now detects and opens supported archives (zip, tar, gz, bz2, xz, 7z with optional extra) when `file_scan.scan_compressed: true`, scanning inner members and saving findings as `archive.zip|inner/path`.
- Share connectors (SMB, WebDAV, SharePoint, NFS via filesystem) reuse the same helper so compressed files downloaded from shares are scanned when the option is on.
- Dashboard and API expose a “Also scan inside compressed files” option (`scan_compressed`), and the CLI has `--scan-compressed` to override config for one-shot runs.
- Resource guards (`max_inner_size`, compressed extensions) and user-facing warnings are in place to avoid archive bombs and unexpected I/O amplification.

- **CNPJ alphanumeric format support (LGPD_CNPJ_ALNUM):**
- Detector patterns now include `LGPD_CNPJ_ALNUM` (alphanumeric CNPJ with the same structure as the legacy numeric format). Detection is **opt-in** via `detection.cnpj_alphanumeric: true` or via `regex_overrides_file` using the example in `config/regex_overrides.example.yaml`.
- `SENSITIVITY_DETECTION*.md` and `USAGE*.md` document CNPJ formats (legacy numeric vs alphanumeric) and how to enable alphanumeric detection, including override-only scenarios.
- New tests validate both patterns across scanner and report flows; the Excel report’s “CNPJ format compatibility” note summarises where legacy, alphanumeric, and mixed formats appear.

- **Dashboard/API scan reliability and feedback:**
- The dashboard Start scan button now surfaces clearer status and error messages (including rate limiting, already-running scan, and auth failures) and disables/enables itself consistently while scans are in progress.
- Non-OK API responses (4xx/5xx) are decoded and displayed instead of failing silently, improving operator visibility when scans are rejected or blocked.

- **Security and readiness groundwork:**
- A `scripts/check-all.ps1` gate was added to run pre-commit (Ruff lint/format) and the full pytest suite in one command.
- `PLAN_READINESS_AND_OPERATIONS.md` now contains explicit reminders to encode secure defaults and production hardening into Cursor rules/skills (loopback host for non-container runs, prod profile requiring API key, no bare try/except/pass, no plaintext passwords in tracked configs).

## Changes since 1.5.4

- **Version:** 1.6.0 in `pyproject.toml`, `core/about.py` fallback, man pages (`docs/data_boar.1`, `docs/data_boar.5`), and docs (README, DEPLOY, TECH_GUIDE, docs/README).
- **Compressed files:** Finalised archive helpers, filesystem/share wiring, config knobs, tests, and docs per `PLAN_COMPRESSED_FILES.md`; plan steps 1–12 are now complete.
- **CNPJ alphanumeric:** Added `LGPD_CNPJ_ALNUM` as a built-in pattern gated by `detection.cnpj_alphanumeric`, plus override examples and tests for both numeric and alnum formats.
- **Dashboard/API:** Improved error handling and user feedback for scan start failures and rate limiting in `api/static/dashboard.js` and `api/routes.py`.
- **Tooling:** Added `scripts/check-all.ps1` as a single entry point for local lint + test checks, to act as a safe gate for Commit/PR even when Agent Review is unavailable or out of sync.

## How to get 1.6.0

- **From source:** `uv sync` or `pip install -e .` after pulling the tag or branch.
- **Docker (branded):** `docker pull fabioleitao/data_boar:1.6.0` or `docker pull fabioleitao/data_boar:latest`

## Building and pushing the Docker image

From the repo root, after tests pass:

```bash
uv run pytest -v -W error --ignore=tests/tests_qa
docker build -t fabioleitao/data_boar:latest -t fabioleitao/data_boar:1.6.0 .
docker login
docker push fabioleitao/data_boar:latest
docker push fabioleitao/data_boar:1.6.0
```

## After merging the release PR (tag, Docker, GitHub release)

1. **Create and push the version tag** (from `main` after merge):

   ```bash
   git checkout main
   git pull origin main
   git tag v1.6.0
   git push origin v1.6.0
   ```

1. **Build and push the Docker image** (see commands above).

1. **Create the GitHub release** (use the notes in this file):

   ```bash
   gh release create v1.6.0 --notes-file docs/releases/1.6.0.md --title "1.6.0"
   ```

   Or in the GitHub web UI: **Releases** → **Draft a new release** → choose tag `v1.6.0`, title `1.6.0`, and paste the contents of `docs/releases/1.6.0.md`.
